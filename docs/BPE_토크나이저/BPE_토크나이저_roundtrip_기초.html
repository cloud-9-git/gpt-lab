BPE Tokenizer 1단계: byte round-trip 먼저 만들기

train과 merge rule을 구현하기 전에, _init_special_tokens, merge 없는 encode, merge 없는 decode만으로 한국어 문장을 원문 그대로 복원하는 단계입니다.

전체 BPE 문서로 돌아가기
1. 왜 이걸 먼저 하나 2. 목표 상태 3. _init_special_tokens 4. merge 없는 encode 5. merge 없는 decode 6. 한국어 복원 확인 7. 디버깅 체크

1. 왜 이 단계를 먼저 하나

BPE tokenizer는 크게 두 층으로 나뉩니다. 첫 번째 층은 모든 문자열을 UTF-8 byte token으로 바꾸고 다시 문자열로 복원하는 기본층입니다. 두 번째 층은 자주 나오는 byte/token pair를 합쳐 token 수를 줄이는 merge층입니다.

merge층은 기본층 위에서만 안전하게 동작합니다. 기본 byte encode/decode가 불안정하면 train에서 merge token을 아무리 잘 만들어도 한국어가 깨지고, 어디서 문제가 생겼는지 추적하기가 어려워집니다.

이 문서의 완료 기준은 train 없이도 decode(encode("한글 English 123")) == "한글 English 123"가 되는 것입니다.

2. 목표 상태

이 단계에서는 self.merges가 비어 있어도 괜찮습니다. 중요한 것은 특수 token과 byte token 256개가 정확히 잡히고, 모든 문자가 byte 단위로 빠짐없이 왕복되는 것입니다.

항목 완료 기준
id_to_token ID 0~3은 특수 token 문자열, ID 4~259는 bytes([0])~bytes([255])
token_to_id id_to_token의 반대 방향 매핑이 모두 들어 있음
merges 아직 빈 리스트여도 됨
encode 문자열을 UTF-8 byte token ID 리스트로 변환
decode byte token ID 리스트를 원래 UTF-8 byte 배열로 복원한 뒤 문자열로 변환

예상 예시

"한".encode("utf-8")
# b'\xed\x95\x9c'

list("한".encode("utf-8"))
# [237, 149, 156]

# BYTE_OFFSET이 4이므로 byte token ID는:
[237 + 4, 149 + 4, 156 + 4]
# [241, 153, 160]
이 단계에서 "한"은 하나의 token이 아닙니다. byte token 3개입니다. BPE 학습이 진행되면 나중에 이 3개가 하나 또는 두 개의 merge token으로 합쳐질 수 있습니다.

3. _init_special_tokens 구현

이 함수는 tokenizer의 기본 vocabulary를 만드는 역할입니다. 이후 train, encode, decode가 모두 이 매핑에 의존합니다.

구현 흐름

  1. self.id_to_token, self.token_to_id, self.merges를 초기 상태로 비웁니다.
  2. SPECIAL_TOKENS를 순회하면서 ID 0~3에 등록합니다.
  3. byte 값 0~255를 순회합니다.
  4. 각 byte의 token ID는 BYTE_OFFSET + byte_value로 계산합니다.
  5. 각 byte token 값은 반드시 bytes([byte_value])로 저장합니다.

의사 코드

def _init_special_tokens(self):
    self.id_to_token = {}
    self.token_to_id = {}
    self.merges = []

    for token, idx in SPECIAL_IDS.items():
        self.id_to_token[idx] = token
        self.token_to_id[token] = idx

    for byte_value in range(NUM_BYTES):
        token_id = BYTE_OFFSET + byte_value
        token = bytes([byte_value])
        self.id_to_token[token_id] = token
        self.token_to_id[token] = token_id

직접 확인

tok = BPETokenizer(vocab_size=300)
tok._init_special_tokens()

assert tok.id_to_token[0] == "<pad>"
assert tok.id_to_token[3] == "<eos>"
assert tok.id_to_token[4] == bytes([0])
assert tok.id_to_token[259] == bytes([255])
assert tok.token_to_id[bytes([65])] == 69
bytes([byte_value])와 byte_value는 다릅니다. id_to_token[69]에 65를 저장하는 게 아니라 b"A"를 저장해야 합니다.

4. merge 없는 encode 구현

아직 merge rule을 적용하지 않는 encode는 매우 단순합니다. 문자열을 UTF-8 byte로 바꾸고, 각 byte 값에 BYTE_OFFSET을 더하면 됩니다.

구현 흐름

  1. 기본 vocabulary가 비어 있으면 _init_special_tokens()를 호출합니다.
  2. text.encode("utf-8")로 byte열을 얻습니다.
  3. 각 byte 값 b를 BYTE_OFFSET + b로 변환합니다.
  4. 이 단계에서는 self.merges가 비어 있으므로 merge 적용은 하지 않아도 됩니다.
  5. add_bos_eos=True이면 앞뒤에 <bos>, <eos> ID를 붙입니다.

의사 코드

def encode(self, text, add_bos_eos=False):
    if not self.id_to_token:
        self._init_special_tokens()

    ids = [BYTE_OFFSET + b for b in text.encode("utf-8")]

    # 지금 단계에서는 self.merges가 비어 있으므로 아무 일도 하지 않음
    # 나중에 train 구현 후 여기에 merge rule 적용 로직이 들어감

    if add_bos_eos:
        ids = [self.get_bos_id()] + ids + [self.get_eos_id()]

    return ids

예시

입력 UTF-8 byte token ID
"A" [65] [69]
"한" [237, 149, 156] [241, 153, 160]
"한A" [237, 149, 156, 65] [241, 153, 160, 69]
byte-level tokenizer는 처음 보는 한국어 단어도 <unk>로 바꾸지 않습니다. 모든 문자는 UTF-8 byte로 표현할 수 있기 때문입니다.

5. merge 없는 decode 구현

decode는 encode의 반대입니다. token ID에서 byte 값을 복원하고, 모든 byte를 모은 뒤 마지막에 한 번만 UTF-8 decode합니다.

구현 흐름

  1. 결과를 담을 byte_values 리스트를 만듭니다.
  2. 입력 ID를 순회합니다.
  3. skip_special=True이고 ID가 0~3이면 건너뜁니다.
  4. ID가 4~259이면 id - BYTE_OFFSET을 byte_values에 추가합니다.
  5. 마지막에 bytes(byte_values).decode("utf-8")를 호출합니다.

의사 코드

def decode(self, ids, skip_special=True):
    byte_values = []

    for token_id in ids:
        if skip_special and token_id in SPECIAL_IDS.values():
            continue

        if BYTE_OFFSET <= token_id < BYTE_OFFSET + NUM_BYTES:
            byte_values.append(token_id - BYTE_OFFSET)
            continue

        # merge token은 다음 단계에서 재귀적으로 펼치도록 구현
        # 지금 단계에서는 encode가 merge token을 만들지 않으므로 도달하지 않음

    return bytes(byte_values).decode("utf-8")

왜 마지막에 한 번만 decode해야 하나

"한"의 byte는 [237, 149, 156]입니다. 이 중 237 하나만 보면 완성된 문자가 아닙니다. 따라서 byte 하나씩 chr로 바꾸거나 하나씩 decode하면 한국어가 깨집니다. 

byte_values = [237, 149, 156]
text = bytes(byte_values).decode("utf-8")
assert text == "한"
chr(token_id), chr(byte_value), bytes([b]).decode(...)를 반복하는 방식은 한국어 복원에 부적합합니다. byte 배열 전체를 만든 뒤 한 번에 decode하세요.

6. 한국어 문장 복원 확인

이 단계에서는 pytest tests/test_bpe.py -v 전체가 아직 통과하지 않아도 됩니다. train, save, load가 남아 있기 때문입니다. 대신 아래 smoke test로 byte round-trip부터 확인합니다. 

from src.bpe import BPETokenizer

tok = BPETokenizer(vocab_size=300)
tok._init_special_tokens()

samples = [
    "이 영화는 정말 좋았다!",
    "연기, 음악, 연출 모두 좋음 ㅋㅋ",
    "한글 English 123 !?",
    "",
]

for text in samples:
    ids = tok.encode(text, add_bos_eos=True)
    restored = tok.decode(ids, skip_special=True)
    assert restored == text, (text, ids, restored)

print("byte-level round-trip passed")

직접 눈으로 확인하기

text = "한글"
ids = tok.encode(text, add_bos_eos=True)
print(ids)
print(tok.decode(ids))

# 예상 형태:
# [2, 241, 153, 160, 238, 188, 132, 3]
# 한글
여기까지 되면 BPE의 바닥공사는 끝난 것입니다. 다음 단계에서 merge token을 추가해도 decode는 결국 byte 배열을 복원한다는 원칙만 유지하면 됩니다.

7. 디버깅 체크

증상 가능한 원인 확인 방법
한국어가 깨진다 byte를 하나씩 문자로 변환함 bytes(byte_values).decode("utf-8")를 마지막에 한 번만 쓰는지 확인
id_to_token[4]가 이상하다 byte token을 bytes([0])가 아니라 int나 문자열로 저장함 assert tok.id_to_token[4] == bytes([0])
<bos>, <eos>가 출력에 섞인다 skip_special 처리가 없음 ID 0~3을 건너뛰는지 확인
빈 문자열에서 에러가 난다 byte 리스트가 비어 있는 경우를 고려하지 않음 bytes([]).decode("utf-8") == "" 확인
테스트에서 ID가 안 맞는다 BYTE_OFFSET을 더하지 않거나 두 번 더함 "A"가 [69]가 되는지 확인

이 단계에서 아직 하지 않아도 되는 것

  • pair_counts를 세는 로직
  • 가장 빈번한 pair를 찾아 새 token으로 만드는 로직
  • self.merges를 encode에 적용하는 로직
  • merge token을 재귀적으로 펼치는 decode 로직
  • JSON save/load
단, 나중에 merge token을 추가할 때도 오늘 만든 원칙은 그대로 유지됩니다. decode는 항상 “token ID를 byte 배열로 펼치고, 마지막에 UTF-8 decode”하는 함수입니다.
