Byte-level BPE Tokenizer 구현 가이드

이 문서는 src/bpe.py의 TODO를 직접 구현하기 위한 학습용 자료입니다. 목표는 외부 tokenizer 없이 UTF-8 byte에서 시작해 BPE merge rule을 학습하고, 한국어 문장을 깨뜨리지 않는 encode/decode를 만드는 것입니다.

1. 목표 2. 핵심 모델 3. ID 설계 4. train 구현 5. encode 구현 6. decode 구현 7. save/load 8. 테스트 체크 초기 round-trip 심화

1. 구현 목표

GPT 모델은 문자열을 바로 입력받지 못합니다. 문자열을 정수 token ID의 리스트로 바꾸고, 모델 출력 token ID를 다시 문자열로 되돌리는 부품이 tokenizer입니다.

이번 과제의 tokenizer는 단어 사전을 외부에서 가져오지 않습니다. 모든 문자를 UTF-8 byte로 쪼갠 뒤, 자주 붙어 나오는 이웃 token pair를 합치면서 vocabulary를 키우는 byte-level BPE 방식입니다.

“vocabulary를 키운다”는 말은 tokenizer가 알고 있는 token 종류를 늘린다는 뜻입니다. 처음에는 <pad>, <unk>, <bos>, <eos> 4개와 byte token 256개만 있습니다. 그래서 시작 vocabulary 크기는 260개입니다. 이후 corpus에서 (241, 153) 같은 pair가 자주 보이면 “앞으로 이 pair는 260번 token 하나로 부르자”라고 새 token을 추가합니다. 그 다음 pair도 추가하면 261, 또 추가하면 262처럼 vocabulary가 커집니다.

여기서 말하는 tokenizer는 “문자열을 숫자로 번역하는 부품”입니다. GPT 모델 안의 embedding layer는 문자열 "한글"을 직접 받을 수 없고, [241, 153, 160, 238, 188, 132] 같은 정수 리스트를 받아야 합니다. 그래서 encode는 문자열을 정수 리스트로 바꾸고, decode는 정수 리스트를 다시 문자열로 되돌립니다.
BPETokenizer.train(corpus)에서의 “학습”은 GPT 모델의 가중치를 학습한다는 뜻이 아닙니다. 여기서는 긴 텍스트 묶음인 corpus를 훑으면서 어떤 byte/token 조합이 자주 붙어 나오는지 세고, 자주 나오는 조합을 새 token으로 등록합니다. 즉 “모델을 똑똑하게 만드는 학습”이 아니라 “tokenizer가 쓸 규칙 목록을 만드는 학습”입니다.
최종적으로 만족해야 하는 가장 중요한 성질은 decode(encode(text)) == text입니다. 특히 한국어에서 이 성질이 깨지면 byte 처리나 merge token 복원이 잘못된 것입니다.

2. 핵심 모델

입력 문자열에서 byte로

text = "한글"
raw_bytes = text.encode("utf-8")
# [237, 149, 156, 234, 184, 128]

사람에게는 두 글자지만 UTF-8에서는 여섯 byte입니다. 이 byte 값에 BYTE_OFFSET을 더한 값이 최초 token ID입니다.

예를 들어 첫 byte 237은 글자 "ㅎ"이 아닙니다. [237, 149, 156] 세 byte가 모두 모였을 때 "한"이 됩니다.

"한글"은 먼저 byte 숫자 [237, 149, 156, 234, 184, 128]이 됩니다. 여기에 BYTE_OFFSET = 4를 각각 더하면 [241, 153, 160, 238, 188, 132]가 됩니다. 이것이 merge를 하나도 적용하지 않았을 때의 가장 기본 token ID 리스트입니다.

BPE 자주 붙는 쌍 합치기

[241, 153, 160, 238, 188, 132]
가장 흔한 pair가 (241, 153)이라면
새 token 260을 만들고 치환

[260, 160, 238, 188, 132]

merge token은 결국 원래 byte token들의 묶음입니다. 따라서 decode에서는 merge token을 byte token까지 펼쳐야 합니다.

260은 "한" 전용 번호가 아닙니다. (241, 153)이라는 pair를 합친 번호입니다. 그래서 같은 pair를 가진 다른 글자에서도 재사용될 수 있습니다.

BPE는 세 개를 한 번에 합치지 않습니다. [241, 153, 160]을 바로 하나로 만드는 대신, 먼저 (241, 153)을 260으로 만들고 그 다음 (260, 160)을 261로 만들 수 있습니다. 그래서 merge token은 이전 merge token을 다시 포함할 수 있습니다.

필요한 내부 자료구조

이름 역할 예시
id_to_token ID에서 token 정보 찾기 {4: b"\x00", 260: (241, 153)}
token_to_id token 정보에서 ID 찾기 {b"\x00": 4, (241, 153): 260}
merges 학습된 merge rule 순서 저장 [(241, 153), (260, 160)]

세 자료구조가 실제로 쓰이는 장면

token_to_id는 encode할 때 주로 씁니다. 예를 들어 현재 숫자 리스트를 왼쪽부터 보다가 (241, 153)이라는 pair를 발견했다고 해봅시다. 이 pair가 학습 때 새 token으로 등록되어 있다면 token_to_id[(241, 153)]를 통해 결과 ID 260을 찾을 수 있습니다. 그래서 encode는 [241, 153, 160]을 [260, 160]처럼 줄일 수 있습니다.

id_to_token은 decode할 때 주로 씁니다. decode는 260을 보고 “이게 원래 뭐였지?”를 알아야 합니다. id_to_token[260]이 (241, 153)이면, decode는 260을 다시 241, 153으로 펼칩니다. merge token이 여러 번 겹쳐 있으면 byte token이 나올 때까지 계속 펼칩니다.

merges는 순서 때문에 필요합니다. BPE는 한 번에 세 개, 네 개를 합치는 방식이 아니라 항상 두 token짜리 pair를 하나로 합칩니다. 그리고 어떤 pair를 먼저 합쳤는지가 다음 merge 결과에 영향을 줍니다. 그래서 encode도 학습 때 만들어진 순서 그대로 merge rule을 적용해야 합니다.

3. Token ID 설계

이 과제는 특수 token ID와 byte token ID가 고정되어 있습니다. 테스트도 이 배치를 기준으로 확인합니다.

ID 범위 의미 저장 형태
0 <pad> 문자열
1 <unk> 문자열
2 <bos> 문자열
3 <eos> 문자열
4~259 원본 byte 0~255 bytes([byte_value])
260+ BPE merge token (left_id, right_id)

특수 token은 왜 필요한가

특수 token은 실제 문장에 들어 있는 글자가 아니라, 모델과 tokenizer가 약속해서 쓰는 표지판입니다. <pad>는 길이를 맞추기 위한 빈칸, <bos>는 시작 표시, <eos>는 끝 표시입니다. <unk>는 모르는 token이라는 뜻이지만, byte-level 방식에서는 모든 문자를 byte로 표현할 수 있어서 자주 쓰이지는 않습니다.

BYTE_OFFSET을 더하는 이유

byte 값은 원래 0~255입니다. 그런데 ID 0~3은 이미 특수 token이 쓰고 있습니다. 그래서 byte 값에 BYTE_OFFSET = 4를 더해서 byte token ID를 4~259로 밀어 둡니다. 

byte 0   -> token ID 4
byte 65  -> token ID 69
byte 237 -> token ID 241
byte 255 -> token ID 259

이 처리를 하지 않으면 문제가 생깁니다. 예를 들어 byte 값 0을 token ID 0으로 쓰면, token ID 0이 <pad>인지 byte 0인지 구분할 수 없습니다. 그래서 byte token은 무조건 특수 token 4개 뒤에서 시작하게 만듭니다.

bytes([byte_value])는 왜 쓰나

byte_value는 0~255 사이의 숫자입니다. bytes([65])는 숫자 65 하나를 담은 byte 객체이고, 화면에는 b"A"처럼 보입니다. 한글의 경우 bytes([237]) 하나만으로는 문자가 되지 않습니다. bytes([237, 149, 156]).decode("utf-8")처럼 세 byte가 모여야 "한"이 됩니다.

_init_special_tokens 구현 순서

  1. id_to_token, token_to_id, merges를 초기화합니다.
  2. 특수 token 4개를 ID 0~3에 넣습니다.
  3. byte 값 0~255를 순회합니다.
  4. ID는 BYTE_OFFSET + byte_value로 정합니다.
  5. token 값은 bytes([byte_value])로 저장합니다.

4. train 구현

BPE 학습은 현재 token ID 시퀀스에서 가장 자주 등장하는 이웃 pair를 찾고, 그 pair를 새 token 하나로 치환하는 일을 반복합니다.

문장을 하나 두고 천천히 보면

문자열: "한글"
UTF-8 byte: [237, 149, 156, 234, 184, 128]
처음 token ID sequence: [241, 153, 160, 238, 188, 132]

이웃 pair들:
(241, 153), (153, 160), (160, 238), (238, 188), (188, 132)

여기서 sequence는 그냥 “순서가 있는 리스트”입니다. 위 예시에서는 [241, 153, 160, 238, 188, 132]가 현재 token ID sequence입니다. 처음에는 token 하나가 byte 하나를 의미합니다. 하지만 merge가 진행되면 token 하나가 여러 byte 묶음을 의미할 수 있습니다.

pair는 이 리스트에서 바로 옆에 붙어 있는 token 두 개입니다. [241, 153, 160] 안에는 (241, 153)과 (153, 160)이라는 pair가 있습니다. train은 corpus 전체에서 이런 pair들이 각각 몇 번 나오는지 셉니다. 가장 자주 나온 pair는 “앞으로 이 둘을 하나의 token으로 취급하자”는 규칙이 됩니다.

전체 흐름

1. 기본 vocabulary 초기화
2. corpus를 UTF-8 bytes로 변환
3. 각 byte를 byte token ID로 변환
4. vocab_size에 도달할 때까지 반복
   4-1. 이웃 pair 빈도 세기
   4-2. 가장 빈도가 높은 pair 선택
   4-3. 새 token ID 생성
   4-4. pair를 새 token ID로 치환
   4-5. merges, id_to_token, token_to_id 갱신

corpus는 tokenizer가 규칙을 배울 때 보는 텍스트 전체입니다. 예를 들어 영화 리뷰 문장 만 개를 하나로 이어 붙인 긴 문자열이 corpus가 될 수 있습니다. BPE는 이 corpus를 먼저 byte token ID sequence로 바꾼 다음, 그 안에서 어떤 pair가 자주 붙어 나오는지 봅니다.

vocab_size는 vocabulary의 목표 크기입니다. vocabulary는 tokenizer가 알고 있는 token 목록입니다. 처음에는 특수 token 4개와 byte token 256개가 있으므로 총 260개에서 시작합니다. vocab_size=300이면 새 merge token을 최대 40개까지 추가할 수 있습니다.

pair 빈도 세기

pair_counts = {}
for a, b in zip(ids, ids[1:]):
    pair = (a, b)
    pair_counts[pair] = pair_counts.get(pair, 0) + 1

zip(ids, ids[1:])는 리스트를 한 칸 밀어서 나란히 보는 코드입니다. 예를 들어 ids = [10, 20, 10, 20]이면 ids는 [10, 20, 10, 20], ids[1:]는 [20, 10, 20]입니다. 둘을 zip으로 묶으면 (10, 20), (20, 10), (10, 20)이 나옵니다. 그래서 (10, 20)의 빈도는 2가 됩니다. 

ids = [10, 20, 10, 20]

이웃 pair:
(10, 20)
(20, 10)
(10, 20)

pair_counts:
(10, 20): 2
(20, 10): 1

pair 치환 아이디어

왼쪽부터 보면서 현재 위치와 다음 위치가 best pair이면 새 token ID 하나를 넣고 두 칸 이동합니다. 아니면 현재 token을 그대로 넣고 한 칸 이동합니다. 

new_ids = []
i = 0
while i < len(ids):
    if i < len(ids) - 1 and (ids[i], ids[i + 1]) == best_pair:
        new_ids.append(new_id)
        i += 2
    else:
        new_ids.append(ids[i])
        i += 1

여기서 두 칸 이동하는 이유가 중요합니다. (10, 20)을 새 token 260으로 바꿨다면, 이미 10과 20 두 token을 소비한 것입니다. 그래서 다음 검사는 그 다음 위치에서 시작해야 합니다. 

best_pair = (10, 20)
new_id = 260

기존: [10, 20, 10, 20]
치환: [260, 260]
성능 최적화는 나중 문제입니다. 먼저 작은 corpus에서 정확히 동작하게 만드세요. corpus[:5000], vocab_size=300 정도로 통과 확인 후 키우는 편이 안전합니다.

5. encode 구현

encode는 학습 때와 같은 방식으로 문자열을 byte token ID로 바꾼 뒤, 저장된 merge rule을 학습 순서대로 적용합니다.

encode는 “새 문장을 모델이 읽을 수 있는 숫자 리스트로 바꾸는 함수”입니다. 학습된 merges가 없다면 byte token ID까지만 만들고 끝납니다. 학습된 merges가 있다면 자주 등장한 조합을 더 큰 token으로 바꿉니다.

중요한 점

  • 처음에는 text.encode("utf-8")를 사용합니다.
  • 각 byte 값은 BYTE_OFFSET + byte_value로 변환합니다.
  • self.merges에 저장된 순서대로 pair 치환을 반복합니다.
  • add_bos_eos=True이면 앞에 2, 뒤에 3을 붙입니다.
"ABAB"
byte token IDs: [69, 70, 69, 70]
merge rule: (69, 70) -> 260
encoded: [260, 260]

merge rule을 적용한다는 말의 의미

처음: [69, 70, 69, 70]
rule: (69, 70) -> 260

왼쪽부터 보면서 (69, 70)을 만나면 260으로 교체

결과: [260, 260]

encode는 새 문장을 볼 때도 train 때 배운 merge rule을 그대로 사용합니다. train 때 (69, 70)을 260으로 합치기로 배웠다면, 나중에 다른 문장에서 똑같은 (69, 70)이 나와도 똑같이 260으로 바꿉니다. 그래서 한 번 학습한 vocabulary와 merges를 저장해 두면, 이후 문장들을 일관된 방식으로 숫자화할 수 있습니다.

encode에서 merge rule을 적용할 때 새 ID는 token_to_id[pair]로 찾을 수 있습니다. pair가 vocabulary에 없다면 그 merge rule은 건너뛰는 방식으로 방어할 수 있습니다.

6. decode 구현

decode는 token ID 리스트를 다시 byte 리스트로 펼친 뒤, 마지막에 한 번만 UTF-8 decode를 해야 합니다.

decode를 “ID 하나를 문자 하나로 바꾸는 함수”라고 생각하면 한국어에서 바로 막힙니다. 정확한 생각은 “ID들을 원래 byte 숫자들로 복원한 다음, byte 전체를 문자열로 바꾸는 함수”입니다.

왜 byte 하나씩 decode하면 안 되나?

한국어 한 글자는 보통 UTF-8 byte 3개입니다. 예를 들어 "한"은 [237, 149, 156]입니다. 중간 byte 하나만 문자로 decode하면 깨질 수 있습니다.

merge token 펼치기

id_to_token[260] = (241, 153)
id_to_token[261] = (260, 160)

261을 펼치면
-> 260, 160
-> 241, 153, 160
-> byte 값으로는 237, 149, 156

위 예시에서 261은 바로 byte가 아닙니다. 261은 (260, 160)을 의미하고, 그 안의 260도 다시 (241, 153)을 의미합니다. 그래서 decode는 ID를 한 번만 바꾸고 끝내면 안 됩니다. byte token ID가 나올 때까지 계속 펼쳐야 합니다.

byte token ID는 4~259 범위입니다. 이 범위의 ID를 만나면 BYTE_OFFSET을 빼서 원래 byte 값으로 되돌립니다. 예를 들어 token ID 241은 byte 값 237입니다. 이렇게 모은 byte 값들이 [237, 149, 156]이 되면, 마지막에 bytes(...).decode("utf-8")로 "한"을 복원합니다.

구현 아이디어

  1. 특수 token은 skip_special=True일 때 건너뜁니다.
  2. ID가 byte token이면 id - BYTE_OFFSET을 byte 리스트에 추가합니다.
  3. ID가 merge token이면 재귀적으로 왼쪽, 오른쪽 token을 펼칩니다.
  4. 마지막에 bytes(byte_values).decode("utf-8", errors="replace")를 호출합니다.
decode의 핵심은 “문자열을 조립한다”가 아니라 “원래 byte 배열을 복원한다”입니다.

7. save / load 구현

JSON은 Python의 bytes와 tuple을 그대로 저장하지 못합니다. 따라서 type 정보를 명시해서 저장하고, load할 때 원래 타입으로 되돌려야 합니다.

왜 저장과 복원이 필요한가

BPE vocabulary를 학습하는 데 시간이 걸립니다. 매번 실행할 때마다 corpus를 다시 보고 pair 빈도를 다시 세면 비효율적입니다. 그래서 한 번 학습한 id_to_token, token_to_id, merges를 파일로 저장해 두고, 다음 실행에서는 load로 바로 복원합니다.

단, JSON 파일은 문자열, 숫자, 리스트, 객체 같은 단순한 값만 안전하게 저장합니다. Python의 bytes와 tuple은 그대로 저장할 수 없어서 {"type": "byte", "value": 65}처럼 풀어서 저장해야 합니다.

예를 들어 Python 안에서는 merge token을 (69, 70) 같은 tuple로 저장할 수 있습니다. 하지만 JSON으로 저장하면 tuple이라는 타입이 사라지고 리스트 [69, 70]처럼 저장됩니다. 그래서 load할 때는 이 리스트를 다시 tuple로 바꿔야 합니다. 마찬가지로 b"A" 같은 bytes 객체도 JSON에 그대로 넣을 수 없으므로, 저장할 때는 {"type": "byte", "value": 65}처럼 숫자로 풀고, load할 때 bytes([65])로 되돌립니다.

저장 형식 예시

{
  "vocab_size": 3000,
  "id_to_token": {
    "0": {"type": "special", "value": "<pad>"},
    "4": {"type": "byte", "value": 0},
    "260": {"type": "merge", "value": [69, 70]}
  },
  "merges": [[69, 70], [260, 71]]
}

load에서 복원할 것

  • id_to_token의 key를 문자열에서 int로 바꿉니다.
  • type == "special"이면 문자열 token으로 복원합니다.
  • type == "byte"이면 bytes([value])로 복원합니다.
  • type == "merge"이면 tuple(value)로 복원합니다.
  • token_to_id는 복원된 id_to_token을 순회하며 다시 만듭니다.
  • merges는 내부 원소를 tuple로 바꿔 [(a, b), ...] 형태로 만듭니다.

tok = BPETokenizer(vocab_size=300)

9. 구현 순서 추천

  1. _init_special_tokens를 먼저 구현합니다.
  2. merge 없이도 동작하는 encode, decode를 만듭니다.
  3. 한국어 문장 복원이 되는지 확인합니다. 이 3단계만 자세히 보는 별도 문서
  4. train에서 pair count와 pair replacement를 구현합니다.
  5. 학습된 merges를 encode에 적용합니다.
  6. save, load를 구현하고 복원 후 encode/decode를 다시 확인합니다.
이 순서로 하면 디버깅이 편합니다. decode가 먼저 안정되어야 train에서 만든 merge token도 안전하게 검증할 수 있습니다.
작성 기준 파일: src/bpe.py. 이 문서는 과제 구현을 돕는 학습 자료이며, 외부 tokenizer 라이브러리를 사용하지 않는 byte-level BPE 구현을 전제로 합니다.
