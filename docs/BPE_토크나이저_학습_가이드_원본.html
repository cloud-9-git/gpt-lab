Byte-level BPE Tokenizer 구현 가이드

이 문서는 src/bpe.py의 TODO를 직접 구현하기 위한 학습용 자료입니다. 목표는 외부 tokenizer 없이 UTF-8 byte에서 시작해 BPE merge rule을 학습하고, 한국어 문장을 깨뜨리지 않는 encode/decode를 만드는 것입니다.

1. 목표 2. 핵심 모델 3. ID 설계 4. train 구현 5. encode 구현 6. decode 구현 7. save/load 8. 테스트 체크 초기 round-trip 심화

1. 구현 목표

GPT 모델은 문자열을 바로 입력받지 못합니다. 문자열을 정수 token ID의 리스트로 바꾸고, 모델 출력 token ID를 다시 문자열로 되돌리는 부품이 tokenizer입니다.

이번 과제의 tokenizer는 단어 사전을 외부에서 가져오지 않습니다. 모든 문자를 UTF-8 byte로 쪼갠 뒤, 자주 붙어 나오는 이웃 token pair를 합치면서 vocabulary를 키우는 byte-level BPE 방식입니다.

최종적으로 만족해야 하는 가장 중요한 성질은 decode(encode(text)) == text입니다. 특히 한국어에서 이 성질이 깨지면 byte 처리나 merge token 복원이 잘못된 것입니다.

2. 핵심 모델

입력 문자열에서 byte로

text = "한글"
raw_bytes = text.encode("utf-8")
# [237, 149, 156, 234, 184, 128]

사람에게는 두 글자지만 UTF-8에서는 여섯 byte입니다. 이 byte 값에 BYTE_OFFSET을 더한 값이 최초 token ID입니다.

BPE 자주 붙는 쌍 합치기

[241, 153, 160, 238, 188, 132]
가장 흔한 pair가 (241, 153)이라면
새 token 260을 만들고 치환

[260, 160, 238, 188, 132]

merge token은 결국 원래 byte token들의 묶음입니다. 따라서 decode에서는 merge token을 byte token까지 펼쳐야 합니다.

필요한 내부 자료구조

이름 역할 예시
id_to_token ID에서 token 정보 찾기 {4: b"\x00", 260: (241, 153)}
token_to_id token 정보에서 ID 찾기 {b"\x00": 4, (241, 153): 260}
merges 학습된 merge rule 순서 저장 [(241, 153), (260, 160)]

3. Token ID 설계

이 과제는 특수 token ID와 byte token ID가 고정되어 있습니다. 테스트도 이 배치를 기준으로 확인합니다.

ID 범위 의미 저장 형태
0 <pad> 문자열
1 <unk> 문자열
2 <bos> 문자열
3 <eos> 문자열
4~259 원본 byte 0~255 bytes([byte_value])
260+ BPE merge token (left_id, right_id)

_init_special_tokens 구현 순서

  1. id_to_token, token_to_id, merges를 초기화합니다.
  2. 특수 token 4개를 ID 0~3에 넣습니다.
  3. byte 값 0~255를 순회합니다.
  4. ID는 BYTE_OFFSET + byte_value로 정합니다.
  5. token 값은 bytes([byte_value])로 저장합니다.

4. train 구현

BPE 학습은 현재 token ID 시퀀스에서 가장 자주 등장하는 이웃 pair를 찾고, 그 pair를 새 token 하나로 치환하는 일을 반복합니다.

전체 흐름

1. 기본 vocabulary 초기화
2. corpus를 UTF-8 bytes로 변환
3. 각 byte를 byte token ID로 변환
4. vocab_size에 도달할 때까지 반복
   4-1. 이웃 pair 빈도 세기
   4-2. 가장 빈도가 높은 pair 선택
   4-3. 새 token ID 생성
   4-4. pair를 새 token ID로 치환
   4-5. merges, id_to_token, token_to_id 갱신

pair 빈도 세기

pair_counts = {}
for a, b in zip(ids, ids[1:]):
    pair = (a, b)
    pair_counts[pair] = pair_counts.get(pair, 0) + 1

pair 치환 아이디어

왼쪽부터 보면서 현재 위치와 다음 위치가 best pair이면 새 token ID 하나를 넣고 두 칸 이동합니다. 아니면 현재 token을 그대로 넣고 한 칸 이동합니다. 

new_ids = []
i = 0
while i < len(ids):
    if i < len(ids) - 1 and (ids[i], ids[i + 1]) == best_pair:
        new_ids.append(new_id)
        i += 2
    else:
        new_ids.append(ids[i])
        i += 1
성능 최적화는 나중 문제입니다. 먼저 작은 corpus에서 정확히 동작하게 만드세요. corpus[:5000], vocab_size=300 정도로 통과 확인 후 키우는 편이 안전합니다.

5. encode 구현

encode는 학습 때와 같은 방식으로 문자열을 byte token ID로 바꾼 뒤, 저장된 merge rule을 학습 순서대로 적용합니다.

중요한 점

  • 처음에는 text.encode("utf-8")를 사용합니다.
  • 각 byte 값은 BYTE_OFFSET + byte_value로 변환합니다.
  • self.merges에 저장된 순서대로 pair 치환을 반복합니다.
  • add_bos_eos=True이면 앞에 2, 뒤에 3을 붙입니다.
"ABAB"
byte token IDs: [69, 70, 69, 70]
merge rule: (69, 70) -> 260
encoded: [260, 260]
encode에서 merge rule을 적용할 때 새 ID는 token_to_id[pair]로 찾을 수 있습니다. pair가 vocabulary에 없다면 그 merge rule은 건너뛰는 방식으로 방어할 수 있습니다.

6. decode 구현

decode는 token ID 리스트를 다시 byte 리스트로 펼친 뒤, 마지막에 한 번만 UTF-8 decode를 해야 합니다.

왜 byte 하나씩 decode하면 안 되나?

한국어 한 글자는 보통 UTF-8 byte 3개입니다. 예를 들어 "한"은 [237, 149, 156]입니다. 중간 byte 하나만 문자로 decode하면 깨질 수 있습니다.

merge token 펼치기

id_to_token[260] = (241, 153)
id_to_token[261] = (260, 160)

261을 펼치면
-> 260, 160
-> 241, 153, 160
-> byte 값으로는 237, 149, 156

구현 아이디어

  1. 특수 token은 skip_special=True일 때 건너뜁니다.
  2. ID가 byte token이면 id - BYTE_OFFSET을 byte 리스트에 추가합니다.
  3. ID가 merge token이면 재귀적으로 왼쪽, 오른쪽 token을 펼칩니다.
  4. 마지막에 bytes(byte_values).decode("utf-8", errors="replace")를 호출합니다.
decode의 핵심은 “문자열을 조립한다”가 아니라 “원래 byte 배열을 복원한다”입니다.

7. save / load 구현

JSON은 Python의 bytes와 tuple을 그대로 저장하지 못합니다. 따라서 type 정보를 명시해서 저장하고, load할 때 원래 타입으로 되돌려야 합니다.

저장 형식 예시

{
  "vocab_size": 3000,
  "id_to_token": {
    "0": {"type": "special", "value": "<pad>"},
    "4": {"type": "byte", "value": 0},
    "260": {"type": "merge", "value": [69, 70]}
  },
  "merges": [[69, 70], [260, 71]]
}

load에서 복원할 것

  • id_to_token의 key를 문자열에서 int로 바꿉니다.
  • type == "special"이면 문자열 token으로 복원합니다.
  • type == "byte"이면 bytes([value])로 복원합니다.
  • type == "merge"이면 tuple(value)로 복원합니다.
  • token_to_id는 복원된 id_to_token을 순회하며 다시 만듭니다.
  • merges는 내부 원소를 tuple로 바꿔 [(a, b), ...] 형태로 만듭니다.

직접 확인용 미니 스크립트

from src.bpe import BPETokenizer

tok = BPETokenizer(vocab_size=300)
tok.train("이 영화는 정말 좋았다. 이 영화는 다시 보고 싶다.")

samples = [
    "이 영화는 정말 좋았다!",
    "연기, 음악, 연출 모두 좋음 ㅋㅋ",
    "한글 English 123 !?",
]

for text in samples:
    ids = tok.encode(text, add_bos_eos=True)
    restored = tok.decode(ids, skip_special=True)
    assert restored == text, (text, restored)

print("BPE tokenizer smoke test passed")

9. 구현 순서 추천

  1. _init_special_tokens를 먼저 구현합니다.
  2. merge 없이도 동작하는 encode, decode를 만듭니다.
  3. 한국어 문장 복원이 되는지 확인합니다. 이 3단계만 자세히 보는 별도 문서
  4. train에서 pair count와 pair replacement를 구현합니다. 이 단계 자세히 보기
  5. 학습된 merges를 encode에 적용합니다. encode 적용 부분 보기
  6. save, load를 구현하고 복원 후 encode/decode를 다시 확인합니다.
이 순서로 하면 디버깅이 편합니다. decode가 먼저 안정되어야 train에서 만든 merge token도 안전하게 검증할 수 있습니다.
